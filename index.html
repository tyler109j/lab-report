📖 What we're doing here — full project dossier

The one-paragraph pitch

This is a self-driving quantitative FX research pipeline. Every 15 minutes it picks 1-3 backtest experiments (strategy + pair + parameter set), runs them against cached OANDA candle data, scores them, updates a Pareto frontier, forward-tests the survivors out-of-sample, walk-forward validates the most promising ones, and re-renders this static HTML dashboard. It is backtest/research only — no live trading. The OANDA API is used read-only for candles; no order, account, or trade endpoints are ever called.

The end state we want: a small set of strategy × pair × parameter combinations that have cleared all 8 promotion gates with stable OOS performance, trading a shared $1,000 paper account with 1% risk per trade and a 30% portfolio drawdown cap. The strategies trade concurrently against the same capital — losing streaks in one reduce the buying power of the others, so the shared-account equity curve shows compounding but with correlated-drawdown risk.

How it actually works — the architecture

  1. Data layer (data.py) — fetches M15 + H1 candles from OANDA practice API, caches them as parquet in data/<PAIR>_<TF>_<N>.parquet. Refreshed daily at 05:00 UTC by daily_refresh_data.sh.
  2. Strategy layer (strategies.py) — 10 strategy families, each a function (df_15m, df_1h, params) → list[signal]. Signals are {side, entry_idx, sl, tp, max_hold, reason}. Param spaces live in PARAM_SPACES; sample_params() / mutate_params() / crossover_params() drive the search.
  3. Engine (engine.py) — walks 15m bars chronologically. At each bar, runs the strategy function on the current data and any open position against current price. Applies spread, swap, and 50% margin closeout. Returns BacktestResult with trades, equity_curve, sharpe, max_dd, etc.
  4. Cycle (cycle.py) — one experiment: pick (strategy, pair, params), run engine, persist to lab.db via db.record_run(), update is_pareto_kept. Random vs. refine vs. crossover vs. robust split per meta-cycle weights.
  5. Forward-test (forward_test.py) — for runs that pass in-sample, re-run against a held-out window (last 30% of data) and compute OOS metrics. Writes oos_* columns.
  6. Walk-forward (rolling_forward.py) — every 6 hours, take the top forward-passers and re-validate against rolling 30/60/90-day windows. Feeds the stability gate.
  7. Eligibility (db.update_eligibility) — runs the 8 promotion gates. A run is live_eligible=1 only if it passes all 8. Currently 10 runs are eligible (all from the confluence family).
  8. Paper track (paper_track.py) — every 15 minutes, takes signals from currently-live runs and simulates fills against recent prices. No OANDA order API. Writes live_signals.
  9. Dashboard (report.py) — assembles this page from DB state. Diff-aware public push via publish.sh → tyler109j/lab-report GitHub Pages.

The 8 promotion gates (the gauntlet)

A run becomes live_eligible=1 only if it clears all 8:

  1. OOS Sharpe ≥ 1.0 (oos_sharpe)
  2. Risk:reward > 1.0 (rr)
  3. Walk-fwd median ≥ 1.0 (wf_median)
  4. Walk-fwd % positive ≥ 0.5 (wf_pct)
  5. Deflated Sharpe > 0.95 (deflated)
  6. OOS trades ≥ 5 (oos_n)
  7. OOS max DD ≤ 30% (oos_dd)
  8. Walk-fwd windows ≥ 3 (wf_n)

Re-run every Sunday 06:00 UTC by weekly_revalidate.sh. Any Pareto run that newly clears all 8 is auto-promoted to live.

Risk model

  • Per-trade risk: 1% of the current shared-account balance at the stop-loss distance.
  • Portfolio drawdown cap: 30% peak-to-trough across the shared $1,000 account. Hard limit — not enforced yet by code, just monitored.
  • Starting account: $1,000 USD (modeled). Engine's internal sizing uses $10k, scaled down by 10× to the shared account.
  • Leverage: 50× (OANDA retail). Fixed 10,000-unit lots per pair per trade.
  • Margin closeout: enforced at 50% equity loss vs. used margin, evaluated every bar.
  • Costs: per-pair spread (0.8-1.6 pips typical for majors) + slippage (~0.2 pips) + swap (triple on Wednesdays). Recorded as costs_version per run; v1 had no costs, v2+ has realistic costs.

Where the data comes from

  • FX candles: OANDA v20 REST API, /v3/instruments/<PAIR>/candles with granularity=M15 and granularity=H1. Read-only against the practice environment using the OANDA API token stored in ~/.config/oanda/token.
  • No proprietary data: no COT reports, no order book data, no sentiment feeds. Only public price action on 15-minute and 1-hour bars.
  • Coverage: 9 OANDA majors only: EUR_USD, GBP_USD, USD_JPY, USD_CHF, AUD_USD, NZD_USD, USD_CAD, EUR_GBP, EUR_JPY. No crosses beyond EUR/GBP and EUR/JPY, no metals, no indices.
  • Lookback: ~187 days of M15 candles as of this render (3,000 bars per pair, fetched daily). Need 1+ year to cover multiple rate-decision regimes.

What the numbers in this dashboard actually mean

  • In-sample (IS) Sharpe: Sharpe ratio computed across all closed trades in the backtest window. Optimistic by construction — this is what the search is optimizing.
  • OOS Sharpe: Sharpe on a held-out window (last 30% of data the strategy never saw during search). Honest signal. Required ≥ 1.0 for promotion.
  • Max DD: peak-to-trough drawdown on the equity curve, expressed as a fraction (0.30 = 30% drawdown). Hard cap: 30% on the shared account.
  • Deflated Sharpe: corrects IS Sharpe for the number of trials attempted (a high Sharpe from 10k random configs is less impressive than the same Sharpe from 10). Lives in the deflated_sharpe column. We display CI in the per-strategy health section.
  • Bootstrap Sharpe CI: 500-resample bootstrap 95% confidence interval on per-trade returns. Narrow CI = stable estimate; wide CI = noisy, even if point estimate looks good.
  • Walk-forward passers: runs where the rolling 30/60/90-day OOS Sharpe stays positive in >50% of windows. Tracks regime stability, not just one-shot OOS luck.

Honest limits (what we don't know yet)

  • Live execution: 0 actual paper fills. 145 signals logged in dry-run; no real OANDA fills, no real spread/slippage confirmation. paper_track.py simulates fills but hasn't been validated against live fills.
  • Costs: v2 spread + swap model hasn't been calibrated against real OANDA fills. Slippage is a fixed 0.2 pips — real execution could be much wider during news.
  • Regimes: the 187-day window is mostly trending FX. No 2020-style vol spike, no central-bank shock, no USD-supply crisis in sample. Robustness across regimes is unproven.
  • Stub families: 3 of 10 strategies (order_flow_imbal, keltner_squeeze, ichimoku_cloud) are scaffolded stubs that return [] signals. They consume rotation slots but produce 0 trades until implemented.
  • Correlated-drawdown risk: all 10 live-eligible runs are the same family (confluence). If that family stops working, the entire shared account feels it.

🧪 Strategy families in rotation

Every day at 04:00 UTC, the system picks the next family from the queue, scaffolds a basic implementation (if not yet done), runs 30 random configs of it through the backtester, and rotates the queue. The 14-entry queue gives us ~2 weeks of new families to explore before any repeat. A family stays in the rotation as long as it's in the cycle, and the cycle weights families with no qualifying runs (n < 30 or Sharpe < 0.5) more heavily — underperformers get more attention.
